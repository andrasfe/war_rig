# CardDemo System Overview

The CardDemo system provides a mechanism for users within a CICS environment to generate transaction reports asynchronously via batch processing. The system allows users to initiate report generation from a CICS screen, offloading the actual report creation to a batch job. This design enables users to continue working within the CICS environment without waiting for the report to be generated, improving overall system responsiveness and efficiency. The system leverages extrapartition transient data queues (TDQs) to facilitate communication between the online CICS environment and the batch processing environment.

The core functionality revolves around receiving user input, validating it, constructing a JCL job to generate the report, and submitting the job to the batch processing system. The system ensures that the report generation process does not tie up valuable CICS resources, allowing for a more scalable and responsive online transaction processing environment.

## Subsystems/Modules

The CardDemo system consists of a single module responsible for initiating batch report generation:

*   **Report Generation Initiation:** This module handles the online interaction with the user, input validation, JCL construction, and job submission.

## Data Flow

1.  The user interacts with a CICS screen (CORPT0AI) to specify the parameters for the transaction report.
2.  The `CORPT00C` program receives the input from the CICS screen and validates it.
3.  `CORPT00C` constructs a JCL job based on the user's input.
4.  The JCL is written to an extrapartition TDQ named `JOBS`. This acts as a trigger for the batch processing system.
5.  The `CORPT00C` program returns control to the previous CICS screen (CORPT0AO).
6.  (Implicit) A separate batch process (not detailed in the provided documentation) reads the JCL from the `JOBS` TDQ and executes the report generation.
